# ADR 0006：糖块从「全局被动加成」改为「投给建筑升级」

- **状态**：已采纳
- **日期**：2026-07-05
- **前置**：[ADR 0003](0003-late-game-and-polish.md)、[ADR 0004](0004-scope-boundary.md)

---

## 背景

ADR 0003 §2 把糖块做成了一个**最小实现**：一次最多一颗未采糖块，30 分钟
成熟，收获即累加，每颗提供**永久 +1% 全局 CPS**（加法叠进
`CurrentCpsRaw` 的 `permanentBonus`）。当时明确留了扩展点：

> 「后续要接原版的 'spend on building level' 时，`long SugarLumps` 已经是
> 余额概念，再加一个 `Dictionary<BuildingId,int> BuildingLevels` 就行」。

本 ADR 记录把这个扩展点补完时的决策，以及一次**重要的数值定位纠偏**。

### 触发这次改动的两个认识

1. **当前糖块没有取舍**。收了自动 +1% 全局，玩家没有任何决策。原版糖块的
   乐趣核心恰恰是「这颗投给哪个建筑」的分配决策 + 收获时的一点惊喜感。
2. **糖块的正确定位是「调味」，不是「成长源」**。原作者把它定位成一个
   *微不足道*的小加成——它不该、也不需要像建筑 / 科技 / 转生那样构成一条
   有实感的成长轴。用「会不会压垮指数曲线」来判断它是**问错了问题**；正确
   的问题是「它有没有超出『微不足道』这个定位」。

按第 2 点的正确基准回看，原来的 30 分钟成熟**严重偏离了定位**（见下）。

---

## 采纳的决策

### 1. 效果模型：投给建筑，每级 +1% 该建筑产出（加法）

**决策**：糖块不再是「收获即 +1% 全局」。改为——

- 收获进入**可花费余额**（`SugarLumps`，语义从「已生效加成计数」变为「未花费余额」）。
- 玩家把糖块**投给某个建筑**升一级：升到第 N 级花 **N 颗**（累计三角数
  `1+2+…+N = N(N+1)/2`）。
- 每一级给**该建筑** +1% 产出，**加法**叠加（Lv10 = 该建筑 ×1.10），乘进
  `GetBuildingUnitCps` 的 per-building `mult`，**不再进 `CurrentCpsRaw` 的全局
  `permanentBonus`**。

**理由**：

- 完全对齐原版：加法（不是 `1.01^N` 复利）、单建筑、三角数花费。
- **数值天然防膨胀，且比旧模型弱得多**。见 §3 的推导：平摊到全部建筑时，
  「投建筑」的全局加成是 **√N 级**（`+√(N/9)%`），而旧的「全局加法」是
  **线性 N 级**（`+N%`）。同样玩一年，新模型比旧模型弱约两个数量级。
- 补齐了唯一的短板——**分配决策**：糖块稀缺时，「投给当前 CPS 占比最高的
  建筑」成为一个真实且可读的最优化小游戏。

**代价**：糖块因此变得**相当弱**（见 §3 数字）。这是「用深度换强度」的
自觉取舍，**符合**糖块「调味而非成长源」的定位，不是缺陷。

### 2. 成熟时间：24 小时（对齐原版量级），推翻 ADR 0003 的 30 分钟

**决策**：`SugarLumpRipenSeconds` 从 30 分钟改为 **24 小时**。

**理由**：

- **公式与成熟时间是一套平衡，必须一起对齐**。原版糖块加成弱，靠的是
  「加法公式 + 约 23 小时成熟」共同压制量级；只搬公式不搬时间，会把一个
  「一年 +6%」的调味系统做成「一年 +44%」的次级成长源（见 §3）。
- **纠正 ADR 0003（以及本轮早期讨论）的一个错误论证**：曾认为「量级由
  三角数花费独立控制，成熟时间可维持 30 分钟」。这是错的——三角数花费只
  压得住「集中投一个建筑」的**次优**玩法；玩家的**最优**策略是**平摊**，而
  平摊在 30 分钟下照样跑到 +44%。**成熟时间才是控制量级的主导杠杆**，花费
  曲线不能替代它。
- **稀缺本身就是「决策」与「惊喜」的来源**。30 分钟一颗 → 糖块廉价、随手
  丢、投哪个都无所谓 → 决策是假的。24 小时一颗 → 一天一颗、每颗都值得
  琢磨 → 决策与惊喜都成立。
- **「一天一颗」是放置游戏最经典的日常钩子**，与本项目的短会话取向不冲突：
  玩家每天回来点一颗、投一次，不需要盯着看。
- 24 小时 = 干净的「一天一颗」节奏；原版 canonical 约 23 小时，取整对齐。

**代价**：老玩家的糖块产出节奏骤降（30 分钟 → 24 小时）。可接受——这正是
把系统拉回其应有定位；且旧的高产出本就是一个应予纠正的偏差，不是既得权益。

### 3. 数值定位：糖块是「调味」，不是「成长源」——附推导

**这是本 ADR 的核心论据，务必保留。**

设玩家攒到 N 颗糖块。最优策略是**平摊到全部 18 个建筑到同一等级 L**——因为
「每建筑 +L%」在全部建筑同级时，数学上等价于「全局 +L%」：

```
总CPS = Σ(建筑ᵢ × (1 + L/100)) = (1 + L/100) × Σ建筑ᵢ
```

平摊到 18 个建筑各 Lv L 的花费 ≈ `18 × L(L+1)/2 ≈ 9L²` 颗。
反解得 **全局加成 ≈ +√(N/9) %**——**平方根级**，不是线性。

完美收获（每 T 小时一颗）玩一年，平摊后的全局加成：

| 成熟时间 | 一年攒到 N | 平摊后每建筑等级 | **一年全局加成** |
| --- | --- | --- | --- |
| **24 小时（采纳）** | ~365 | Lv 6 | **≈ +6%** |
| 20 小时（原版量级） | ~438 | Lv 6~7 | ≈ +6.5% |
| 4 小时 | ~2190 | Lv 15 | ≈ +15% |
| 1 小时 | ~8760 | Lv 31 | ≈ +31% |
| 30 分钟（旧值，已废弃） | ~17520 | Lv 44 | ≈ +44% |

对比：**旧的「全局加法」模型**同样一年（17520 颗）是 **+17520%（×176）**。

结论：
- 24 小时 + 投建筑，让糖块回到「一年约 +6%」的**微不足道调味**定位。
- 判断糖块的正确基准是**它自身的定位**（调味），不是拿它比建筑 / 科技 / 转生
  这些成长源。之前反复用「相对指数经济可忽略」来为 30 分钟辩护，是**用错了
  基准线**。

### 4. 存档迁移 v4 → v5：老糖块原样退成余额（策略 A）

**决策**：`SaveData` 加字段；`SugarLumps` 字段**语义转变**为「未花费余额」。

- 老存档里累计的 `SugarLumps` **原样保留为余额**，玩家自行重新分配到建筑。
- 新增 `BuildingLevels: Dictionary<BuildingId,int>`，老存档默认空。
- 迁移时给玩家一条新闻提示：「你的 N 颗糖块现在可以投给建筑升级了」。

**理由**：

- 策略 A（退成余额）远比策略 B（自动折算成建筑等级）简单、可解释，且不亏
  玩家——老玩家一颗不少，只是要手动投一次，正好体验新的分配决策。
- 语义转变（「已生效加成」→「未花费余额」）必须在迁移注释里写明，这是
  本次 schema 变更**最容易误解**的一点。
- 遵循 ADR 0003 §8 的 `while(Version < Current)` 步进迁移结构；v4→v5 case
  只做「保留 SugarLumps 当余额 + BuildingLevels 默认空 + 版本号 +1」。

**代价**：老玩家重开后会发现「糖块加成暂时归零、变成一堆待分配余额」，需靠
新闻提示 + UI 引导说明。一次性的认知成本，可接受。

### 5. UI：独立 modal 做分配

**决策**：糖块分配用一个**独立 modal**（参考现有 Options modal 模式），不塞进
`SugarLumpWidget` 就地展开。

- Widget 保留：成熟进度条 + 收获按钮 + 余额 `🍬 N`，新增一个「分配」入口。
- Modal 内是一个建筑列表，每行 `建筑名 · Lv X · [升级（花 X+1 颗）]`，余额
  不足则按钮禁用。列表数据全来自 `Buildings.All` + `BuildingLevels`，零硬编码。
- blurb / 文案改为说明「投给建筑升级，每级 +1% 该建筑产出」，中 / 繁 / 英三份。

**理由**：24 小时一颗意味着分配是个**低频、需要斟酌**的动作，配独立 modal 比
「widget 里挤一排小按钮」更适合「停下来想一想投哪个」的节奏。

## 主动没做的事

- **稀有度（Bifurcated / Golden / Meaty…）**：不做。它是 RNG + 与
  Grandmapocalypse / wrinkler / 季节活动**耦合**的机制，而这些正是 ADR 0004
  §2–§3 明确拒绝的系统。没有那些系统，稀有度只是「有名字没内涵」。
- **糖块门控后期系统 / 建筑等级解锁小游戏**：不做，同属 ADR 0004 拒绝范围。
- **成熟速度加速链**（原版靠升级 / 特定糖块类型缩短下一颗时间）：不做，
  它服务于原版的超长线滚雪球，与本项目定位无关。
- **成熟度窗口（原版的 20h 长成 / 23h 成熟 / 24h 落下）**：不做。原版糖块有一条
  三段时间线——20 小时「长成」后提前采收产出打折，23 小时「成熟」是最佳采收点，
  超过 24 小时未采则**自动掉落**（可能损失或变异）。这套机制的本质是**奖励玩家在
  正确的时间点回来采收**——一种主动注意力管理，正是 ADR 0004 §2 拒绝的「上班打卡」
  性质。本项目的模型刻意更宽容：糖块成熟后**一直等玩家点，不打折、不流失**，
  符合 idle「放着就好」的核心诉求。我们只保留「成熟 → 采收 → 投建筑」这条干净
  主线，不引入时间窗口带来的风险 / 惩罚 / 时机博弈。

## 已知的限制 / 债务

- **糖块偏弱是设计结论，不是待修 bug**（见 §1 代价、§3 数字）。未来若有人
  觉得「糖块没存在感」想上调，请先回到 §3 的定位论证：它**本就该**是调味。
  要改强度，改的应是「定位」这个前提，那需要新 ADR，而不是悄悄调数值 / 时间。
- 与**离线收益**的交互：ADR 0003 §4 的 `ApplyOfflineProgress` 会推进糖块成熟
  计时。24 小时成熟 + 24 小时离线上限意味着离线最多正好催熟一颗——与「一天
  一颗」的节奏自洽，无需特殊处理，但实现时要写测试固定这个行为。
- 分配是**不可逆**的（投出去的糖块不能撤回重投）。原版亦如此。这本身就是
  「决策有分量」的一部分，不视为债务。

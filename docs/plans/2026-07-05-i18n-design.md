# 多语言支持设计（简体 / 繁体 / 英文）

日期：2026-07-05

## 目标

为游戏增加多语言支持，暂时支持**简体中文、繁体中文、英文**三种语言。首次进入时
自动检测系统语言并应用，玩家可在设置中手动切换。

## 已确认的关键决策

| 决策点 | 结论 | 理由 |
| --- | --- | --- |
| 覆盖范围 | **全覆盖**：界面文案 + 领域内容（建筑/升级/成就/新闻） | 既然要做就做彻底，key 化架构无论如何都要建 |
| 翻译数据落点 | **放在 `Game.Core`**，Core 持有翻译 catalog 与语言状态 | 符合"所有游戏内容在 Core"、可被 xUnit 直接测试的既有架构；遵守 Core 零 Blazor 依赖铁律 |
| 繁体是否动态生成 | **否，两份手写翻译，共用同一套 key** | 简→繁存在一对多歧义（面→麵/面、发→發/髮）、需 OpenCC 级词库、地道用词差异；本项目文本量不大，手写更可控 |
| 数字显示格式 | **保持英文单位（million/billion/…），不本地化** | 中文缺乏成熟大数词汇（兆/京母语者无概念），英文大数体系成熟通用；与原版一致 |
| 成就名数量级词 | **保持英文**（"Trillion 烘焙师"） | 与"数字用英文"一致，玩家在数字面板与成就里概念对得上 |
| 语言字段可空性 | **可空 `Language?`**：`null`=跟随系统，非空=玩家已选 | 能区分"跟随系统"与"固定某语言"，符合"自动检测 + 玩家可选"需求 |
| 通知队列 | **存 key + 参数**，UI 出队时翻译 | 彻底多语言，切换语言后历史新闻也正确；不让 Core 碰 localizer 成品字符串 |

## 整体架构

遵守铁律：**`Game.Core` 零 Blazor/DOM 依赖，UI → domain 单向**。翻译数据和语言
状态都放在 Core，Web 层只负责：检测/设置语言、读取已翻译好的字符串。

**翻译只发生在"要显示给玩家"的显示边界**。`GameState` 领域逻辑继续用 key 和 ID
运转，不注入 localizer，因此领域逻辑仍可无语言地测试。

### 新增组件（均在 `Game.Core/Localization/`）

1. **`Language.cs`** — enum：`SimplifiedChinese`、`TraditionalChinese`、`English`。
2. **`ILocalizer.cs`** — 查询抽象：
   - `string this[string key]` — 按 key 取当前语言字符串；
   - `string Format(string key, params object[] args)` — 模板 + 占位符（成就名等程序化文本）；
   - `Language Current { get; }`、`void SetLanguage(Language lang)`；
   - `event Action OnLanguageChanged`。
3. **`Localizer.cs`** — `ILocalizer` 实现，持有当前语言 + 三张翻译字典，是全局单一真相源。
4. **`TranslationsZhHans.cs` / `TranslationsZhHant.cs` / `TranslationsEn.cs`** — 三份手写
   翻译 catalog，各暴露一个 `IReadOnlyDictionary<string,string>`，风格对齐现有 `Data/` 静态 catalog。
   英文那份从现有 catalog 内联文本迁移而来。

## key 命名规范

点分层级，与现有 catalog ID 风格一致，便于按前缀校验完整性：

```
building.cursor.name / building.cursor.desc
upgrade.<id>.name / upgrade.<id>.desc
achievement.<id>.name / achievement.<id>.desc
achievement.baker.name        (模板，如 "{0} 烘焙师")
ui.upgrades.title / ui.upgrades.count / ui.upgrades.empty
ui.effect.building_multiplier (半程序化 UI 模板，如 "×{0} {1} 产出")
ui.settings.language / ui.save.export ...
news.<语义名>                 (如 news.grandma_awakening)
magnitude.trillion            (数量级词，三语言值相同，保持英文)
```

- **key 由 id 推导**：`BuildingId.Cursor` → `building.cursor.*`，减少 catalog 改动面、
  杜绝拼错。升级、成就同理。
- **新闻 flavor 用语义名**（`news.grandma_awakening`）而非序号，翻译时可读性更好。

## 数据流与显示边界

- **catalog 层（`Data/`）**：`BuildingDefinition`/`UpgradeDefinition`/`AchievementDefinition`
  去掉内联 `Name`/`Description` 字符串。key 由 id 推导；成就的模板 key + magnitude 参数存在 def 上。
- **显示读取**：给 catalog def 加扩展方法 `def.DisplayName(localizer)` / `def.DisplayDesc(localizer)`，
  封装 key 推导 + Format。成就名走 `Format("achievement.baker.name", localizer["magnitude.trillion"])`。
- **半程序化 UI 文本**（如 `UpgradeStore.razor` 的 `EffectText`）：改用模板 key
  `ui.effect.building_multiplier = "×{0} {1} 产出"`，参数填数值与已翻译的建筑名，归在 `ui.effect.*`。
- **`GameState` 领域逻辑**：完全不动，不注入 localizer。唯一变化是通知队列
  改 enqueue **key + 参数** 而非成品英文句子；`DrainNewsMessages()`/成就通知返回 key，UI 出队时翻译。

## 语言检测、切换与持久化

- **自动检测**（首次进入、存档语言为 `null` 时）：Web 层经 JS interop 读 `navigator.language` 映射：
  - `zh-CN` / `zh-SG` / `zh-Hans*` → 简体
  - `zh-TW` / `zh-HK` / `zh-MO` / `zh-Hant*` → 繁体
  - 其它 → 英文（兜底）

  检测在 Web 层完成（需碰 `navigator`），得到 `Language` 后调用 Core 的 `Localizer.SetLanguage()`。
- **手动选择**：设置里提供三选一下拉，选择后立即 `SetLanguage` 并触发全局重渲染。
- **持久化**：`SaveData` 增加可空 `Language?` 字段。`null`=从未手动选过（走自动检测）；
  非空=玩家选过（尊重玩家，不再自动覆盖）。
- **版本迁移**：`SaveData.CurrentVersion` **3 → 4**，`Migrate()` 加 `v3→v4` case——
  旧存档无语言字段，迁移后为 `null`（跟随系统）。
- **重渲染**：`Localizer.SetLanguage()` 触发 `OnLanguageChanged` 事件（类似
  `GameLoop.OnTick`/`OnStateReplaced`），`Home.razor` 订阅并 `StateHasChanged()`。
- **DI**：`Localizer` 在 Web 层注册为单例（与 `SaveCoordinator` 同级），类型定义在 Core。

## 测试策略（`Game.Core.Tests`，全部可测，无需浏览器）

1. **完整性测试（最关键）**：遍历所有建筑/升级/成就 id 推导 key + 所有 `ui.*`/`news.*` key，
   断言三份翻译字典都不缺 key、无空值。防"加了新建筑忘翻译"。
2. **无孤儿 key**：三份字典 key 集合完全一致（一一对应），防某语言多出/拼错。
3. **模板参数测试**：`Format("achievement.baker.name", "Trillion")` 三语言正确套入，`{0}` 不残留。
4. **迁移测试**：v3 存档迁移到 v4 后 `Language == null`；已选语言的存档往返序列化保持不变。
5. **数量级词一致**：`magnitude.*` 三语言值相同且为英文。

## 实施顺序（每步可独立编译 + 测试）

1. 建 `Localization/`：`Language` enum、`ILocalizer`、`Localizer` + `OnLanguageChanged`。
2. 建三份翻译字典，英文从现有 catalog 内联文本迁移。
3. key 化 `Data/` catalog（建筑 → 升级 → 成就 → 新闻），加 `DisplayName/DisplayDesc(localizer)` 扩展。
4. `SaveData` v3→v4：加可空 `Language`、迁移 case。
5. Web 层：DI 注册 `Localizer` 单例、JS 检测 `navigator.language`、`Home.razor` 订阅 `OnLanguageChanged`、语言下拉 UI。
6. 逐个组件把内联英文替换为 localizer 查询（含 `EffectText` 模板化）。
7. `GameState` 通知队列改存 key + 参数，UI 出队翻译。
8. 补全上述测试。

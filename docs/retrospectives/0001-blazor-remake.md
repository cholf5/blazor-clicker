# 复盘 0001 — Blazor 版 Cookie Clicker 一次性交付

- 日期：2026-07-04
- 范围：M1（Domain/Formulas/GameState）→ M2（升级）→ M3（成就）→ M4（存档）+ 金饼干 + UI + 部署工作流
- 结果：约 20 分钟对话内一次性交付可玩版本，实机验证时"完全能玩"，无阻断性 bug
- 参与者：用户（决策 + 实机验证）、AI 助手（实现）

## TL;DR

这次的交付质量高于我们平常对"AI one-shot"的预期，但这**不是"AI 一次写对"**的胜利，而是
下列几个具体条件叠加的结果。如果丢掉其中任意一条，节奏都会明显变慢。本篇的价值不在于
纪念这次做得好，而在于**把配方拆出来，让下次能复现**。

## 真正起作用的东西

### 1. 原版是现成的规格书

原版 Cookie Clicker 的 `main.js`（3000 行的浏览器脚本）虽然结构混乱，但价格公式
（`baseCost × 1.15ⁿ`）、CPS 计算、tier upgrade 阈值（1 / 5 / 25 / 50）、金饼干
概率、Frenzy 倍率与时长等**全是可对着看的数值**。本次 AI 没有在"设计游戏"，而是在
"照着行为对齐 + 组织代码"（原版源码仅作个人参考，未在本仓库分发，见 `NOTICE.md`）。

- 这是 remake 项目独有的红利
- 从零设计的项目享受不到
- 下次做原创游戏时，需要预估"参数设计"本身也是不小的时间投入

### 2. 决策提前收敛

用户在动手前把技术栈选型（Plan A vs D vs F）、目录结构（"blazor 别放根目录"）、
双方案取舍（"两套都试试，先 Blazor"）都拍板了。**AI 没有在写代码时纠结架构**。

- ADR 写在动手之前不是仪式感，是真省时间
- 一旦让 AI 边写边挑技术栈，20 分钟连骨架都搭不完

### 3. 数据驱动 + 静态目录

Buildings（12）/ Upgrades（约 55）/ Achievements（约 60）全是 `record` +
`IReadOnlyList<T>` + 一次性铺开的字面量。

- 这类"表格代码"是 AI 最擅长的形态
- 比复杂控制流可靠得多
- Tier 升级用 `(threshold, cost, prefix)` 元组表批量生成，避免了手写 48 条重复条目

### 4. 测试先行卡住了真 bug

33 个 xUnit 测试覆盖 Formulas、GameState、购买、成就、存档 round-trip、金饼干。
以下几个 bug 只有测试能发现，肉眼看 UI 看不出来：

- Tier upgrade 倍率乘错位置（作用于错误的 building）
- Thousand Fingers "每个非光标建筑加成到光标" 的加数
- 存档 round-trip 时枚举序列化格式
- 未来版本号存档没有被拒绝

**这是敢直接交付的底气**。

### 5. Blazor 组件模型帮了大忙

Scoped CSS（`.razor.css`）、`EventCallback<T>`、DI 注册、`_Imports.razor`——
这些约定让 UI 层几乎没有决策空间，照抄模式就行。

## 没那么神的地方（诚实清单）

### 1. 完全没做运行时冒烟

AI 全程没在浏览器里跑过一次。是**用户在 Rider F5 帮忙兜的底**。

如果下列任一情况发生，AI 是不会知道的：

- 金饼干那段 JS interop 有问题
- LocalStorage key 冲突或存档反序列化炸掉
- CSS 在真实浏览器里错位
- Base href / SPA 路由在 Pages 上 404

**教训**：build/test/publish 干净 ≠ 能玩。下次至少要跑一次 `dotnet run` 手动点几下。

### 2. GameLoop 的引用 bug 是自己复查发现的

第一版 `GameLoop` 缓存了 `GameState` 引用。但 `SaveCoordinator` 在 Import/Wipe 后会
替换 State——旧引用还在，游戏会静默卡死在旧存档上。

修复：改成读 `_save.State.Tick(delta)`，并加 `OnStateReplaced` 事件让 UI 刷新。

**教训**：涉及"可替换聚合根"时，别在别的服务里缓存它。这类 bug 单测很难覆盖，
需要主动画数据流图才发现。

### 3. Offline earnings 被"用文档合理化偷懒"

ADR 0002 把 offline earnings 列为"非目标"。**一部分原因确实是范围控制，但另一部分
原因是不想在没有实机验证的情况下改 tick 的时间逻辑**。

这是"用文档给偷懒背书"的例子。诚实标记出来，避免下次自欺欺人。

### 4. 数字类型选了 `double`

后期升级堆到几百次会到 1e308 并 `Infinity`。原版有 BigNumber，我们没有。
ADR 0002 承认了，但值得再强调：**这不是"暂不实现"，而是"数值上限被硬编码在类型里"**，
一旦要玩到深度后期就必须重写整层 CPS/成本计算。

### 5. 完全没考虑 i18n

所有字符串硬编码在 C# 里。想加中文/其他语言时会很痛。

## 复现配方

想让下次 AI 交付也达到这个质量，需要**同时**满足：

1. **有参考实现**（原版代码 / 竞品 / 详细数值文档）
2. **技术栈提前锁定**，用 ADR 定死，动手前不再改
3. **主逻辑能被建模成表格**（数据驱动 > 控制流）
4. **测试覆盖数值/规则**，UI 交给人工冒烟
5. **用框架默认约定**，别自创模式（DI、组件、CSS 都用官方推荐的方式）
6. **人 + AI 明确分工**：AI 写代码 + 测试；人做实机验证 + 决策拍板

**缺一个都会慢下来**。原创游戏 + 未定技术栈 + 大量运行时状态机 + 无测试文化 +
自研框架——这五条任一命中，"20 分钟一次性交付"就不可能。

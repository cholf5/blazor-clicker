## 何时新增 ADR

- 引入新的核心库/框架
- 改变项目分层或数据流方向
- 决定支持/放弃某个平台或浏览器
- 存档格式/版本迁移策略确定时
- 切换 Plan A ↔ Plan B 时

小规模选型（比如"用哪个 CSS 库"）可以不写 ADR，直接在 PR 描述或代码注释里
说明理由即可。**判断标准**：如果半年后回看会问"为什么当初选它？"就写。

## 何时新增复盘

- 完成一个里程碑（例如 M1–M4 一次性交付、部署上线、切换到 Plan B）
- 出现一次值得记忆的成功或失败（原因不明的高效交付、罕见的返工、重大事故）

复盘的价值是**下次能复现**，不是留念，所以务必包含"没起作用的地方"和
"复现配方"两个部分，避免只写好话。
